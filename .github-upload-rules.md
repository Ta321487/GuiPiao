# GitHub 上传规则配置

## 指令格式
/github 提交信息

## 示例
/github 修复了卡片视图的内存泄漏
/github 优化了地图模块的性能
/github 修复了登录bug

## 版本号规则（智能递增）

### 版本号格式
- 主版本：0.9.x（基础版本）
- 补丁版本：0.9.x.y（小修复）

### 版本号判断逻辑
1. **自动分析**：读取上一个版本的提交信息，提取关键词
2. **主题匹配**：对比本次提交信息的关键词
   - ✅ **完全相同主题**：补丁版本 +1（0.9.15 → 0.9.15.1 → 0.9.15.2）
   - 🔄 **包含新内容/新主题**：主版本 +1（0.9.15 → 0.9.16）
   - 🔄 **混合内容**：包含旧补丁 + 新内容 → 主版本 +1
   - 🔄 **描述模糊**：如只写"修复了bug" → 保守处理 → 主版本 +1

### 关键词提取示例
- "修复卡片" vs "修复卡片视图" → 相同主题（补丁版本）
- "修复卡片" vs "优化地图" → 不同主题（主版本递增）
- "修复卡片；修复内存" vs "修复卡片；新增图表" → **混合内容**（新内容 → 主版本递增）
- "修复卡片" vs "修复了bug" → **描述模糊**（保守 → 主版本递增）

### 最佳实践
- 尽量在提交信息中**明确写出修改的模块/功能**
- 例如："修复卡片视图的显示bug" 而不是 "修复了bug"

## 手动指定版本号
如果AI判断不准确，你可以手动指定：
```
/github 0.9.16 修复了xxx
```

## 自动处理流程
1. 读取最近提交信息（ver 0.9.x.y 描述）
2. 提取关键词（卡片、地图、内存、登录等）
3. 对比本次提交信息的关键词
4. 判断版本号递增方式
5. 生成建议版本号
6. 执行 git add → commit → push
